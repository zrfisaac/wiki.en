# Design Patterns

This documentation covers several essential design patterns commonly used in software development. Each section explains a specific pattern and provides code examples to demonstrate how they work.

## Table of Contents

- [Design Patterns - Adapter](patterns-adapter)  
  A pattern that allows incompatible interfaces to work together by converting one interface into another.

- [Design Patterns - Builder](patterns-builder)  
  A pattern that separates the construction of a complex object from its representation, allowing for more flexible object creation.

- [Design Patterns - Command](patterns-command)  
  A pattern that encapsulates a request as an object, allowing parameterization of clients with queues, requests, and operations.

- [Design Patterns - Decorator](patterns-decorator)  
  A pattern that allows for the dynamic addition of responsibilities to an object without modifying its structure.

- [Design Patterns - Facade](patterns-facade)  
  A pattern that provides a simplified interface to a complex subsystem, hiding the system's complexity.

- [Design Patterns - Factory Method](patterns-factory)  
  A pattern that defines an interface for creating objects, but allows subclasses to alter the type of objects that will be created.

- [Design Patterns - Observer](patterns-observer)  
  A pattern that defines a one-to-many dependency between objects, where a change in one object triggers updates to its dependents.

- [Design Patterns - Singleton](patterns-singleton)  
  A pattern that ensures a class has only one instance and provides a global point of access to that instance.

- [Design Patterns - Strategy](patterns-strategy)  
  A pattern that allows a family of algorithms to be defined and encapsulated, making them interchangeable without altering the clients.

- [Design Patterns - Template Method](patterns-template)  
  A pattern that defines the structure of an algorithm, allowing specific steps of the algorithm to be implemented by subclasses.
